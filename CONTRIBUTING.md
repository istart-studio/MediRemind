# 贡献指南

感谢您考虑为MediRemind项目做出贡献！以下是一些指导方针，以帮助您为项目贡献代码。

## 行为准则

参与此项目的所有贡献者都应遵守我们的行为准则。请尊重所有参与者，并创建一个积极和包容的环境。

## 如何贡献

### 报告Bug

如果您发现了Bug，请通过GitHub Issues报告，并确保：

1. 搜索现有问题，避免创建重复的问题
2. 使用清晰的标题和详细描述
3. 包括重现步骤、预期行为和实际行为
4. 添加相关的截图或错误日志
5. 说明您使用的操作系统和应用版本

### 提出新功能

如果您想提出新功能或改进，请：

1. 通过GitHub Issues提交功能请求
2. 清晰描述该功能将如何工作
3. 解释为什么该功能对大多数用户有用

### 提交Pull Request

1. Fork仓库并创建您的分支
2. 确保代码符合项目的代码规范
3. 确保所有测试通过
4. 提交有意义的commit消息
5. 更新文档（如果需要）
6. 提交Pull Request并详细描述您的更改

## 开发流程

### 环境设置

请按照README.md中的安装指南设置您的开发环境。

### 代码规范

本项目遵循[JavaScript Standard Style](https://standardjs.com/)。在提交代码前，请确保您的代码符合这一规范。

您可以运行以下命令检查代码风格：

```bash
npm run lint
# 或
yarn lint
```

### 提交消息规范

我们使用[约定式提交](https://www.conventionalcommits.org/)规范来格式化提交消息。每个提交消息应该包含一个类型和一个主题：

```
<类型>: <描述>
```

类型可以是：

- `feat`: 新功能
- `fix`: Bug修复
- `docs`: 文档更改
- `style`: 不影响代码含义的更改（空格、格式化等）
- `refactor`: 既不修复错误也不添加功能的代码更改
- `perf`: 提高性能的代码更改
- `test`: 添加或修正测试
- `chore`: 对构建过程或辅助工具的更改

例如：
```
feat: 添加用药提醒功能
fix: 修复日期选择器在Android上的显示问题
```

### 分支策略

- `master`: 主分支，只接受来自`develop`分支的合并
- `develop`: 开发分支，所有功能分支都从这里分出并合并回来
- `feature/*`: 功能分支，用于开发新功能
- `bugfix/*`: 修复分支，用于修复Bug
- `hotfix/*`: 热修复分支，用于紧急修复生产环境中的问题

### 版本发布流程

我们使用[语义化版本](https://semver.org/)进行版本控制：

- 主版本号：当你做了不兼容的API更改
- 次版本号：当你添加了向下兼容的功能
- 修订号：当你做了向下兼容的Bug修复

## 代码审查

所有提交将经过代码审查。审查者将关注：

- 功能性：代码是否按预期工作
- 代码质量：代码是否清晰、简洁、易于维护
- 性能：代码是否高效
- 测试覆盖：是否有足够的测试
